00
LABORA
Electronic Requisition Creation
At the time of specimen collection, an electronic requisition is created. This is the primary data quality gate — missing required fields block submission entirely.
Required Fields
  • Patient demographics: Last name, First name, DOB, Gender, Address (City / State / ZIP)
  • Primary insurance name + Policy number; Secondary if applicable
  • Ordering provider: Full name + NPI
  • ICD-10 diagnosis code(s): minimum 1 — cannot be left blank
  • Ordered tests: Panel code(s); test–specimen type confirmed
  • Billing type: Insurance / Bill Client / Cash / Card — explicit selection required
  • Collection date and location
⊘ Block Gate
  • ICD-10 diagnosis code missing
  • NPI absent
  • Primary insurance missing (Insurance / SNF type)
  • Billing type not selected
After Submission
  • All data pushed directly from LABORA into LABDAQ — no re-entry at accession
  • LABORA scope limited to intake; all further processing in LABDAQ
Added
  • Electronic REQ with mandatory field validation
  • Billing type set at intake
  • Missing ICD/NPI/Insurance blocked here — never reach DM
  • Direct LABDAQ push from LABORA
Removed
  • Paper REQ as sole source
  • Data problems discovered at CS or Billing
  • No intake validation
  • Problem Log Excel for data errors

LABDAQ
Specimen Accession & Verification
Specimen physical integrity is verified and LABDAQ data (populated from LABORA) is confirmed. This step is verification — not data entry.
Accession Actions
  • Specimen integrity assessed: volume, condition, tube type, match to ordered tests
  • LABDAQ data confirmed — corrections made directly in LABDAQ for any discrepancies
  • Classified: Ready for Testing / Problem Panel / Cancel
  • ORGANIZATION_TYPE confirmed per client: Facility / Physician / B2B
◆ Problem Panel — Specimen Issue
  • Definition: Specimen cannot be tested due to physical quality (hemolysis, insufficient volume, wrong tube, degraded sample) — distinct from data problems blocked at Step 00
  • Flagged in LABDAQ — visible via DM filter; no separate Excel Problem Log
  • Client contacted for re-collection → re-enters at Step 00 if successful
  • Not resolved within 7 days → order cancelled; REQ closed in LABDAQ
Lab QA/QC — Billing Eligibility Trigger
  • QA/QC performed on specimen and result
  • Upon completion: REQ_PRINTED_DATE set in LABDAQ — formal billing eligibility trigger
  • Cases without REQ_PRINTED_DATE not extracted in DM
Added
  • Problem Panel classified in LABDAQ; DM-filterable
  • Re-collection protocol (7-day limit)
  • ORGANIZATION_TYPE per client
  • REQ_PRINTED_DATE as billing trigger
Removed
  • Manual data entry from paper REQ
  • Separate Excel Problem Log
  • Data errors detected here (now blocked at Step 00)

02
LABDAQ / DataMiner
Daily DM Extraction — Single Source of Truth
Daily DataMiner export filtered by REQ_PRINTED_DATE. Send-out and Problem Panel cases identifiable directly from DM — no separate Excel logs maintained.
Extraction Filter Logic
  • Extract all records where REQ_PRINTED_DATE is populated
  • Insurance eligible: REQ_PRINTED_DATE present AND PANEL_LAB_NAME = "Open Healthcare"
  • Send-out: PANEL_LAB_NAME ≠ "Open Healthcare" → identified at extraction; routed to Send out only sheet; no separate log
  • Problem Panel: excluded via LABDAQ flag; no separate log
  • Duplicates → DUPES REMOVED sheet (not deleted)
  • SSN always left blank; COMPARE sheet maintained
Expected AR Recognition
  • Expected revenue recognized for all billing-eligible cases at extraction (accrual basis)
  • Calculated using Price Group configured in LABDAQ per client / per test
  • Expected AR entered into QuickBooks — billing is collection mechanism, not revenue creation
⚠ Snapshot Lock

Snapshot used for billing/AR must be locked after extraction. Subsequent LABDAQ changes tracked in next day's COMPARE sheet only.
